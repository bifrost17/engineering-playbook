# Code Reviews

> Reference: [Google Engineering Practices - Code Review](https://google.github.io/eng-practices/review/)

## 프로세스

- **리뷰어 수:** 1명 필수, 2명 상한
- **턴어라운드:** 24시간 이내 (비즈니스 아워 기준)
- **첫 응답:** 리뷰 요청 후 4시간 이내가 이상적
- **스타일/포맷팅:** 100% 자동화 — 사람이 지적하지 않음

## 리뷰 확인 사항 (우선순위순)

1. **설계/아키텍처** — 이 변경이 올바른 위치에 있는가?
2. **기능 정확성** — 의도대로 동작하는가?
3. **복잡도** — 다른 개발자가 이해 가능한가?
4. **테스트** — 적절한 테스트가 포함되었는가?
5. **네이밍** — 변수/함수명이 명확한가?
6. **보안** — 입력 검증, 인증/인가, 시크릿 노출 여부

> 스타일은 목록에 없다. 도구에 위임한다.

## 리뷰어 가이드

### Do

- 질문형으로 피드백 ("이렇게 하면 어떨까요?" > "이건 틀렸습니다")
- 칭찬도 함께 (좋은 코드에는 "nice!")
- 구체적으로 지적 (파일명:라인 번호 + 대안 제시)
- 전체 방향 먼저 확인 후 세부 사항 논의

### Don't

- 스타일 논쟁 (린터가 해결)
- 취향 강요 ("나라면 이렇게 했을 것" → 동작하면 OK)
- 코멘트 폭탄 → 핵심 3-5개에 집중

## 자동화

| 단계 | 도구 | 역할 |
|------|------|------|
| Pre-commit | husky + lint-staged | 포맷팅, 린팅 |
| CI | 린터 + 테스트 + 타입 체크 | 자동 게이트 |
| AI 리뷰 (선택) | CodeRabbit, Copilot Code Review | 초기 리뷰 보조 |

**기본값: 보류.** 소규모 프로젝트에서는 사람 리뷰로 충분. 팀 규모가 커지고 리뷰 부담이 늘면 CodeRabbit 등 도입 검토.
