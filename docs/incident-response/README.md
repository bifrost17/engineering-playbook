# Incident Response

> Reference: [Google SRE Book](https://sre.google/sre-book/managing-incidents/), [PagerDuty Incident Response](https://response.pagerduty.com/)

## 심각도 레벨

| 레벨 | 정의 | 대응 시간 | 예시 |
|------|------|----------|------|
| **P0** | 전체 서비스 중단, 데이터 손실 위험 | 즉시 (15분 이내) | 프로덕션 DB 다운, 전체 접속 불가 |
| **P1** | 주요 기능 장애, 다수 사용자 영향 | 30분 이내 | 결제 불가, API 50% 실패 |
| **P2** | 부분 기능 장애, 우회 가능 | 업무시간 내 | 특정 페이지 느림 |
| **P3** | 경미한 이슈 | 다음 스프린트 | UI 깨짐 |
| **P4** | 개선 사항 | 백로그 | 성능 최적화 가능 지점 |

## 인시던트 대응 프로세스

### 1. 감지 (Detect)

- 모니터링 알림 또는 사용자 보고
- 알림 채널에서 심각도 판단

### 2. 대응 팀 소집 (Mobilize)

- P0/P1: 즉시 전용 Slack 채널 개설 (`#incident-YYYY-MM-DD`)
- **역할 분리:**
  - **Incident Commander (IC):** 조율, 커뮤니케이션 담당
  - **Tech Lead:** 기술적 진단 및 수정
  - **Communicator:** 외부/내부 상태 업데이트

### 3. 진단 및 수정 (Mitigate)

- 런북 참조하여 단계별 진단
- 복구 우선 — 근본 원인 분석은 사후에
- 변경 사항은 모두 기록

### 4. 커뮤니케이션 (Communicate)

- **내부:** P0은 30분마다, P1은 1시간마다 상태 공유
- **외부:** 상태 페이지 업데이트

### 5. 복구 확인 (Resolve)

- SLI/SLO 정상 복귀 확인
- 인시던트 채널에 해결 선언
- 포스트모템 일정 잡기

## 비난 없는 포스트모템 (Blameless Postmortem)

> "사람이 아닌 시스템을 고친다" — Google, Netflix, Etsy 공통 원칙

**모든 P0/P1 인시던트에 필수 작성.**

### 포스트모템 구조

```markdown
# 포스트모템: [인시던트 제목]

## 요약
무엇이 발생했는가 (1-2문장)

## 영향
- 영향받은 사용자 수:
- 지속 시간:
- 매출 영향:

## 타임라인
| 시각 | 이벤트 |
|------|--------|
| HH:MM | 알림 발생 |
| HH:MM | IC 소집 |
| HH:MM | 원인 파악 |
| HH:MM | 수정 배포 |
| HH:MM | 복구 확인 |

## 근본 원인
5 Whys 분석

## 조치 항목 (Action Items)
| 항목 | 담당자 | 기한 | 상태 |
|------|--------|------|------|

## 교훈
### 잘한 점
### 개선할 점
```

### 규칙

- **72시간 이내 작성** (기억이 신선할 때)
- **팀/조직 공유** — 학습 문화 형성
- **조치 항목 추적** — 이슈 트래커에 등록, 완료까지 관리

## 온콜 (On-Call)

- **로테이션:** 주 단위, 최소 2명 (primary + secondary)
- **보상:** 수당 또는 대체 휴무 (번아웃 방지)
- **에스컬레이션 경로:** 1차 → 2차 → 매니저 → VP
- **핸드오프:** 진행 중인 이슈 인계 문서화

기존 툴체인 연동과 예산에 따라 선택. **기본값: Grafana OnCall** (OSS, 무료, Grafana 스택과 통합). 규모 커지면 PagerDuty 또는 Opsgenie 전환.

## 런북 (Runbook)

모든 서비스에 런북 작성 필수:

1. **서비스 개요** — 아키텍처, 의존성
2. **알림 설명** — 이 알림이 왜 발생하는가
3. **진단 단계** — 확인할 메트릭/로그/명령어 (copy-paste 가능)
4. **조치 단계** — 복구 절차
5. **에스컬레이션** — 연락처
6. **이전 인시던트** — 관련 과거 사례 링크

팀 워크플로우에 따라 선택. **기본값: Git repo** (코드와 함께 버전 관리, Documentation-as-Code 원칙 일관성).
