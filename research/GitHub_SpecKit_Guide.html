01 Spec Kit이란?

GitHub Spec Kit은 GitHub이 만든 오픈소스 Spec-Driven Development(SDD) 툴킷입니다. 전통적인 개발 방식(코드 먼저 → 문서 나중)을 뒤집어, 명세(Specification)를 먼저 작성하고, AI 에이전트가 명세를 기반으로 구현하는 방법론입니다.

핵심 철학

"Specifications don't serve code — code serves specifications."

명세가 코드를 위해 존재하는 것이 아니라, 코드가 명세를 위해 존재한다.

왜 Spec Kit인가?

  • Vibe Coding의 한계 극복 — 속도는 유지하되, "실제로 맞는 걸 만들었나?"라는 불확실성을 제거
  • 명세 = 소스 오브 트루스 — 요구사항이 변경되면 명세를 수정하고 구현을 재생성
  • 에이전트 비의존적 — Claude Code, Copilot, Gemini, Cursor 등 23개 이상의 AI 에이전트 지원
  • 기존 프로젝트에도 적용 가능 — Greenfield(신규)와 Brownfield(기존) 프로젝트 모두 지원

Spec-Driven Development vs 기존 방식

구분 기존 방식 Spec-Driven (Spec Kit)
순서 코드 작성 → 문서화 명세 작성 → AI가 구현
소스 오브 트루스 코드 명세 문서
요구사항 변경 코드 수동 수정 명세 수정 → 재생성
AI 역할 코드 자동완성 명세 기반 체계적 구현
품질 보증 코드 리뷰 중심 명세 게이트 + 자동 검증

02 설치 방법

사전 요구사항

  • Python 3.11+
  • Git
  • uv 패키지 매니저
  • Claude Code (AI 에이전트)

방법 1: 영구 설치 (권장)

# 특정 버전 설치 (안정적)
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git@v0.5.2

# 최신 개발 버전 설치
uv tool install specify-cli --from git+https://github.com/github/spec-kit.git

방법 2: 1회성 실행 (설치 없이)

uvx --from git+https://github.com/github/spec-kit.git@v0.5.2 specify init my-project

방법 3: pip 사용

pip install specify-cli --index-url git+https://github.com/github/spec-kit.git

방법 4: 오프라인/에어갭 환경

# 인터넷 연결된 머신에서
pip download specify-cli -d ./dist

# dist/ 폴더를 전송 후
pip install --no-index --find-links=./dist specify-cli

설치 확인

specify check
TIP

uv가 없다면: curl -LsSf https://astral.sh/uv/install.sh | sh 로 설치

03 프로젝트 초기화

기본 초기화 (Claude Code용)

# 새 프로젝트 디렉토리 생성 + 초기화
specify init my-project --ai claude

# 현재 디렉토리에서 초기화 (기존 프로젝트에 적용)
specify init . --ai claude
# 또는
specify init --here --ai claude

주요 옵션

옵션 설명 예시
--ai <agent> AI 에이전트 선택 --ai claude
--script <type> 스크립트 유형 (bash/ps) --script sh
--here 현재 디렉토리에 초기화 specify init --here
--force 기존 파일 덮어쓰기 (업그레이드) specify init --here --force
--ignore-agent-tools CLI 도구 검증 생략
주의

--ai를 생략하면 23개 이상의 에이전트 중 선택하는 인터랙티브 메뉴가 나타납니다. Claude Code를 사용하려면 반드시 --ai claude를 지정하세요.

초기화가 완료되면 Claude Code를 해당 디렉토리에서 실행하면 /speckit.* 슬래시 커맨드가 자동으로 인식됩니다.

06 Phase 1: Constitution (헌법)

PHASE 1 프로젝트 원칙 수립
명령어/speckit.constitution
입력없음 (대화형으로 생성)
출력.specify/memory/constitution.md
목적프로젝트의 불변 원칙을 정의. 이후 모든 단계에서 이 원칙을 기준으로 검증됨

헌법에 포함되는 내용

  1. 5가지 핵심 원칙 — 프로젝트를 관통하는 비타협 규칙
  2. 추가 제약조건 — 기술 요구사항, 컴플라이언스, 배포 제약
  3. 개발 워크플로우 — 코드 리뷰, 승인 게이트 등
  4. 거버넌스 조항 — 헌법이 모든 다른 관행보다 우선

실행 예시

# Claude Code에서
/speckit.constitution

Claude가 대화형으로 프로젝트 목적, 기술 스택, 원칙 등을 질문하고 constitution.md를 생성합니다.

Constitution 예시

# Constitution: Claude Desktop Client

## Core Principles

### 1. API-First Architecture
모든 기능은 Claude API를 통해 구현한다. 자체 AI 모델이나 프록시 없이
Anthropic Messages API를 직접 사용한다.

### 2. Security-First
API 키는 절대로 클라이언트에 노출하지 않는다.
서버 프록시를 통해 모든 API 통신을 처리한다.

### 3. Test-First
모든 기능은 테스트 코드를 먼저 작성한 후 구현한다.

### 4. Type Safety
TypeScript strict mode를 사용하며, any 타입 사용을 금지한다.

### 5. Streaming Default
모든 AI 응답은 스트리밍을 기본으로 한다.
사용자에게 실시간 피드백을 제공한다.

## Additional Constraints
- Framework: Next.js 14+ App Router
- Styling: Tailwind CSS + shadcn/ui
- State: Zustand
- Node.js 20+

## Governance
이 헌법은 모든 설계 및 구현 결정에 우선합니다.
중요

Constitution 위반은 분석 시 CRITICAL 심각도로 분류되며, 해결될 때까지 구현이 중단됩니다. 반드시 신중하게 작성하세요.

07 Phase 2: Specification (명세)

PHASE 2 기능 명세 작성
명령어/speckit.specify
입력기능에 대한 자연어 설명
출력specs/NNN-feature/spec.md, specs/NNN-feature/checklists/requirements.md
목적WHAT(무엇을)과 WHY(왜)를 기술. HOW(어떻게)는 포함하지 않음

실행 예시

# Claude Code에서
/speckit.specify

# Claude가 물을 것:
# "어떤 기능을 명세하시겠습니까?"
# → "Claude 데스크톱 앱과 동일한 채팅 인터페이스.
#    사이드바에 대화 목록, 메인 영역에 채팅, 하단에 입력창.
#    마크다운 렌더링, 코드 하이라이팅, 스트리밍 응답 지원."

생성되는 spec.md의 필수 섹션

섹션 설명 예시
User Stories 우선순위별 사용자 스토리 (P1/P2/P3) Given/When/Then 형식
Functional Requirements 기능 요구사항 (FR-001~) FR-001: 사용자는 새 대화를 시작할 수 있다
Success Criteria 측정 가능한 성공 기준 (SC-001~) SC-001: 메시지 전송 후 1초 이내 응답 시작
Key Entities 핵심 데이터 모델 Conversation, Message, User
Edge Cases 예외 상황 네트워크 끊김, 토큰 초과, Rate limit
Assumptions 전제 조건 사용자는 유효한 API 키를 보유
핵심 규칙

명세에는 구현 상세를 포함하지 않습니다. 언어, 프레임워크, 데이터베이스, 라이브러리 이름 등은 Plan 단계에서 결정합니다. 명세는 오직 "무엇을", "왜"만 기술합니다.

프로세스 상세

  1. Short Name 생성 — 기능 설명에서 2~4단어 키워드 추출 (예: chat-interface)
  2. Git Branch 생성 — (선택) spec/chat-interface 브랜치 생성
  3. 디렉토리 생성 — specs/001-chat-interface/ 자동 생성
  4. 템플릿 채우기 — 대화형으로 spec-template.md 작성
  5. 미확인 항목 — 최대 3개의 [NEEDS CLARIFICATION] 마커 허용
  6. 품질 검증 — 요구사항 체크리스트 생성, 최대 3회 반복 검증

08 Phase 3: Clarification (명확화)

PHASE 3 — 선택 명세 모호성 해결
명령어/speckit.clarify
입력기존 spec.md
출력업데이트된 spec.md + ## Clarifications 섹션 추가
목적명세의 모호한 부분을 발견하고 구체화

분석하는 9가지 카테고리

  1. 기능 범위 (Functional Scope)
  2. 도메인/데이터 모델 (Domain/Data Model)
  3. UX 플로우 (UX Flow)
  4. 비기능 속성 (Non-Functional Attributes)
  5. 통합 (Integration)
  6. 엣지 케이스 (Edge Cases)
  7. 제약조건 (Constraints)
  8. 용어 (Terminology)
  9. 완료 신호 (Completion Signals)

실행 과정

# Claude Code에서
/speckit.clarify

# Claude가 최대 5개의 질문을 한 번에 하나씩 제시
# → 객관식(2~5개 선택지) 또는 단답형(5단어 이하)
# → 각 답변 후 즉시 spec.md 해당 섹션 업데이트

# 조기 종료: "done", "stop", "proceed" 입력
TIP

Plan 단계 전에 Clarify를 실행하면, 구현 중 발견되는 모호성을 크게 줄일 수 있습니다. 특히 복잡한 기능일수록 이 단계를 건너뛰지 마세요.

Phase 1: Design (설계)

  • 데이터 모델 생성
  • 인터페이스 계약 정의
  • 아키텍처 결정 문서화

Phase 2: Planning (계획 수립)

  • 구현 계획 생성
  • 브랜치, 계획 경로, 산출물 보고

PHASE 6 코드 구현 실행
명령어/speckit.implement
입력 (필수)tasks.md + plan.md
입력 (선택)data-model.md, contracts/, research.md
출력실제 소스 코드 + 업데이트된 tasks.md (완료 마킹)
목적태스크를 순서대로 실행하여 코드를 생성

실행 과정

  1. 사전 검증 — 필수 파일(tasks.md, plan.md) 존재 확인
  2. 체크리스트 확인 — checklists/ 폴더가 있으면 완료율 검증
  3. 프로젝트 셋업 — .gitignore, .eslintignore 등 자동 생성
  4. 태스크 실행 — 단계별(Setup → Tests → Core → Integration → Polish) 순서
  5. 진행 추적 — 완료된 태스크를 tasks.md에서 [X]로 마킹
  6. 에러 처리 — 비병렬 태스크 실패 시 중단, 병렬 태스크는 계속 진행

실행 예시

# Claude Code에서
/speckit.implement

# Claude가 tasks.md를 읽고 순서대로 구현 시작
# → T001: Next.js 프로젝트 초기화... ✅
# → T002: Tailwind 설정... ✅
# → T003: Zustand 스토어 설정... ✅
# → T004: API 프록시 라우트 생성... ✅
# ...
# → 각 태스크 완료 시 tasks.md에 [X] 마킹
TDD 방식

Spec Kit은 Test-First를 권장합니다. 태스크 순서가 "테스트 작성 → 구현" 순으로 되어 있으면, Claude가 테스트를 먼저 작성한 뒤 통과하는 구현 코드를 생성합니다.

/speckit.checklist — 품질 게이트

구현 전 명세 완성도를 검증하는 체크리스트를 생성합니다.

체크리스트 항목 형식

- [ ] CHK001 - Are error handling behaviors defined for network timeout scenarios? [Completeness, FR-005]
- [ ] CHK002 - Is "fast response" quantified with specific latency criteria? [Measurability, SC-001]

품질 차원

  • Completeness — 모든 시나리오가 정의되었는가
  • Clarity — 표현이 명확한가
  • Consistency — 문서 간 일관성 있는가
  • Measurability — 객관적 측정이 가능한가
  • Scenario Coverage — 시나리오가 충분한가
  • Edge Cases — 예외 상황이 고려되었는가
  • Non-Functional — 성능/보안/접근성 등
  • Dependencies — 의존성이 명시되었는가
  • Ambiguities — 모호한 부분이 남아있는가

15 Best Practices

1. Constitution을 먼저

반드시 /speckit.constitution을 가장 먼저 실행하세요. Constitution은 이후 모든 단계의 품질 게이트입니다. 여기서 원칙을 명확히 세우면 나중에 돌아가는 일이 줄어듭니다.

2. 명세에 구현 상세를 넣지 마세요

Specification 단계에서 "React", "PostgreSQL", "Next.js" 같은 기술 이름을 언급하지 마세요. 명세는 오직 무엇을(WHAT), 왜(WHY)만 기술합니다. 기술 선택은 Plan 단계에서 합니다.

3. Clarify를 건너뛰지 마세요

특히 복잡한 기능일수록 /speckit.clarify를 반드시 실행하세요. 구현 중 발견되는 모호성을 미리 제거할 수 있습니다.

4. Analyze로 교차 검증

Plan과 Tasks를 생성한 후, 구현 전에 /speckit.analyze를 실행하세요. spec ↔ plan ↔ tasks 간의 누락이나 불일치를 감지합니다.

5. 기능 단위로 분리

하나의 specs/NNN-feature/ 디렉토리에는 하나의 기능만. 큰 프로젝트는 여러 기능 디렉토리로 나누세요.

6. [P] 마커를 적극 활용

독립적인 태스크에 병렬 마커를 붙이면 구현 속도가 크게 향상됩니다. 다른 유저 스토리의 태스크는 기본적으로 병렬 가능합니다.

7. 태스크에 정확한 파일 경로를

모든 태스크 설명에는 정확한 파일 경로를 포함하세요. Implement user service ✗ → Implement UserService in src/services/user.ts ✓

8. 업그레이드 전 백업

Spec Kit을 업그레이드할 때 .specify/memory/constitution.md와 커스텀 템플릿이 덮어씌워질 수 있습니다. 반드시 백업 후 업그레이드하세요.

9. 한 기능씩 완전히 끝내기

하나의 기능에 대해 Constitution → Specify → Clarify → Plan → Tasks → Implement 전체 사이클을 완료한 뒤, 다음 기능으로 넘어가세요.

10. Git 브랜치 전략

각 기능 명세 시 자동으로 브랜치를 생성할 수 있습니다. spec/chat-interface → feat/chat-interface 패턴으로 관리하면 깔끔합니다.
