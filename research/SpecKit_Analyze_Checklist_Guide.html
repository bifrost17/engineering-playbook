◆ 1. /speckit.analyze 개요

/speckit.analyze는 spec.md, plan.md, tasks.md 세 문서와 constitution.md를 교차 검증하는 읽기 전용 분석 도구입니다. 어떤 파일도 직접 수정하지 않고 문제점만 보고합니다.

한 줄 요약

spec ↔ plan ↔ tasks 사이의 누락, 모순, 중복, 헌법 위반을 자동으로 찾아내는 "교차 검증 리포트"

명령어/speckit.analyze
입력spec.md + plan.md + tasks.md + constitution.md (자동 로드)
출력Findings Table + Coverage Summary + Metrics + Next Actions (마크다운 리포트)
파일 수정없음 (읽기 전용)

◆ 3. 5가지 검출 패스

패스 검출 내용 예시
1. Duplication
중복 검출		 중복된 요구사항이나 태스크 FR-003과 FR-007이 동일한 "대화 목록 표시" 기능을 기술
2. Ambiguity
모호성 식별		 정량화되지 않은 모호한 표현 "빠른 응답", "적절한 크기", "충분한 간격" 같은 표현
3. Underspecification
상세 부족		 구현에 필요하지만 정의되지 않은 영역 에러 핸들링 시나리오 미정의, 로딩 상태 미기술
4. Constitution Alignment
헌법 정합성		 Constitution 원칙 위반 여부 Test-First 원칙인데 테스트 태스크가 구현 태스크보다 뒤에 배치됨
5. Coverage Gap
커버리지 누락		 요구사항 대비 태스크 매핑 누락 FR-005(에러 핸들링)에 대응하는 태스크가 tasks.md에 없음

◆ 4. 심각도 분류

심각도 기준 조치
CRITICAL Constitution 위반
핵심 기능(P1) 커버리지 0%		 즉시 수정 필수. 이 상태로 구현 진행 불가. 헌법 원칙은 비타협
HIGH 중요 요구사항 누락
데이터 모델과 태스크 불일치
P1 스토리 커버리지 부분 누락		 구현 전 수정 강력 권고. 무시하면 구현 중 큰 재작업 가능성
MEDIUM 모호한 표현 / 부분적 커버리지
P2/P3 스토리 관련 이슈		 수정 권고. 구현은 가능하지만 품질 저하 위험
LOW 사소한 불일치 / 스타일 이슈
용어 비일관성		 선택적 수정. 무시해도 구현에 큰 영향 없음
절대 규칙

Constitution 위반은 어떤 경우에도 CRITICAL로 분류됩니다. 헌법 원칙은 비타협이며, CRITICAL이 1개라도 있으면 구현 진행 전 반드시 해결해야 합니다.

◆ 5. 출력 예시 (전체 리포트)

아래는 "Claude Desktop Client - Chat Interface" 기능을 분석한 실제 출력 형태입니다.

## Analysis Report: 001-chat-interface

### Findings

| ID    | Category          | Severity | Location      | Summary                                          | Recommendation                         |
|-------|-------------------|----------|---------------|--------------------------------------------------|----------------------------------------|
| A-001 | Constitution      | CRITICAL | tasks.md T007 | Test-First 위반: MessageBubble 구현이 테스트보다 앞 | T007과 T006 순서 교체 필요               |
| A-002 | Coverage Gap      | HIGH     | FR-005        | 네트워크 에러 핸들링 요구사항에 대응 태스크 없음      | tasks.md에 에러 핸들링 태스크 추가         |
| A-003 | Ambiguity         | MEDIUM   | spec.md SC-001| "빠른 응답"의 정량적 기준 미정의                    | "1초 이내 첫 토큰 수신"으로 구체화         |
| A-004 | Underspecification| MEDIUM   | spec.md EC    | Rate limit(429) 발생 시 UX 동작 미정의             | Edge Cases에 Rate limit 시나리오 추가     |
| A-005 | Duplication       | LOW      | FR-003/FR-008 | "대화 목록 표시" 기능이 두 곳에 중복 기술            | FR-008을 FR-003에 통합                   |

### Coverage Summary

- Total Requirements: 12
- Mapped to Tasks: 10
- Coverage: 83.3%
- Unmapped: FR-005 (에러 핸들링), FR-011 (키보드 단축키)

### Metrics

- Findings: 5 (1 CRITICAL, 1 HIGH, 2 MEDIUM, 1 LOW)
- Constitution Compliance: FAIL (1 violation)

### Next Actions

1. 🔴 CRITICAL: tasks.md에서 T006(테스트)과 T007(구현) 순서를 교정하세요
2. 🟠 HIGH: FR-005에 대응하는 에러 핸들링 태스크를 추가하세요
3. 🟡 MEDIUM: spec.md의 SC-001을 정량적 기준으로 구체화하세요
4. 🟡 MEDIUM: Edge Cases에 Rate limit 시나리오를 추가하세요
5. 🔵 LOW: FR-003과 FR-008의 중복을 정리하세요

출력 구조 해설

섹션 내용
Findings Table 발견된 모든 이슈를 ID, 카테고리, 심각도, 위치, 요약, 권고사항으로 정리. 최대 50행
Coverage Summary 전체 요구사항 수 대비 태스크에 매핑된 비율. 미매핑된 요구사항 ID 표시
Metrics 심각도별 이슈 수, Constitution 준수 여부 (PASS/FAIL)
Next Actions 심각도 순으로 정렬된 구체적 수정 지시. 이 목록을 그대로 Claude에게 전달 가능

# - tasks.md에 에러 핸들링 태스크 추가                  → A-002 해결
# - spec.md의 SC-001 정량적 기준으로 구체화              → A-003 해결

방법 2: 전체 수정

# 모든 이슈 일괄 수정
"분석 결과의 모든 이슈를 수정해줘"

방법 3: 심각도 기반 수정

# CRITICAL과 HIGH만 우선 수정
"CRITICAL과 HIGH 이슈만 먼저 수정해줘"

재검증

# 수정 후 다시 분석하여 결과 확인
/speckit.analyze

# → CRITICAL 0개, Coverage 100%가 되면 구현 진행 가능
실전 팁

수정 → 재분석 사이클을 CRITICAL이 0이 될 때까지 반복하세요. HIGH도 가능하면 해결하되, MEDIUM/LOW는 구현 후 보완해도 됩니다.

◆ 7. 핵심 제약사항

제약 설명
읽기 전용 어떤 파일도 직접 수정하지 않음. 분석 리포트만 출력
결정론적 문서 변경 없이 재실행하면 동일한 결과를 생성
최대 50행 토큰 효율성을 위해 Findings Table은 최대 50행. 고신호(high-signal) 이슈에 집중
Constitution 절대 권위 헌법 위반은 어떤 상황에서도 CRITICAL. 예외 없음
Extension Hooks extensions.yml에 before_analyze/after_analyze hook이 있으면 자동 실행

◆ 8. /speckit.checklist 개요

/speckit.checklist는 "영어(명세)를 위한 단위 테스트"를 생성합니다. 코드 테스트가 아니라, 명세 문서 자체의 품질을 검증하는 체크리스트입니다.

한 줄 요약

"요구사항이 충분히 명확하고, 완전하고, 측정 가능한가?"를 체크하는 "명세 품질 게이트"

명령어/speckit.checklist
입력spec.md, plan.md, tasks.md (자동 로드)
출력specs/NNN-feature/checklists/[domain].md
파일 수정새 파일 생성 또는 기존 파일에 추가 (기존 항목 삭제 안 함)

◆ 9. 핵심 개념: 구현 검증 vs 명세 검증

Checklist가 다른 테스트 도구와 구별되는 핵심 차이점입니다. 코드가 동작하는지가 아니라, 명세가 충분한지를 검증합니다.

✗ 잘못된 예 (구현 검증 — 체크리스트가 아님)

"Verify the button displays correctly"

"Test hover states work on desktop"

"Confirm the page renders 3 cards"

"Click the submit button and check response"

금지 동사: Verify, Test, Confirm, Click, Navigate, Render, Load, Execute

✓ 올바른 예 (명세 검증 — 이것이 체크리스트)

"Are visual hierarchy requirements defined with measurable criteria?"

"Is the term 'prominent' quantified with specific sizing?"

"Are keyboard navigation requirements documented for all interactive elements?"

"Are error recovery behaviors specified for network timeout scenarios?"

올바른 패턴: "Are [X] defined/specified for [Y]?", "Is [term] quantified with...?"

◆ 10. 실행 과정 (3단계)

Step 1: 의도 파악 질문 (최대 3개)

Claude가 체크리스트 범위를 좁히기 위해 2~3개 질문을 합니다.

# Claude Code에서
/speckit.checklist

# Claude의 질문 예시:
Claude: 체크리스트의 초점을 맞추기 위해 몇 가지 질문이 있습니다.

1. 어떤 영역을 중점적으로 검증하시겠습니까?
   a) UX/접근성
   b) 보안
   c) 성능
   d) 전체 (broad coverage)

2. 이 체크리스트의 용도는?
   a) 구현 전 최종 게이트 (필수 통과)
   b) 명세 검토 참고용 (가이드라인)

3. 대상 독자는?
   a) 명세 작성자 (본인)
   b) 리뷰어
   c) QA 엔지니어

Step 2: 컨텍스트 로드 & 분석

spec.md, plan.md, tasks.md를 분석하여 9가지 품질 차원에서 체크 항목을 도출합니다.

품질 차원검증 내용
Completeness모든 시나리오가 정의되었는가
Clarity표현이 명확하고 정량적인가
Consistency문서 간 일관성이 있는가
Measurability객관적으로 측정 가능한가
Scenario Coverage사용 시나리오가 충분한가
Edge Cases예외 상황이 고려되었는가
Non-Functional성능, 보안, 접근성 등이 정의되었는가
Dependencies외부 의존성이 명시되었는가
Ambiguities모호한 부분이 남아있는가

Step 3: 체크리스트 파일 생성

# 생성 위치
specs/001-chat-interface/checklists/ux.md         # UX 도메인
specs/001-chat-interface/checklists/security.md    # 보안 도메인
specs/001-chat-interface/checklists/requirements.md # 전체 요구사항

# 항목 형식
- [ ] CHK### - Are [요구사항 측면] defined/specified for [시나리오]? [품질 차원, 참조]

◆ 11. 출력 예시: UX 체크리스트

# UX Checklist: 001-chat-interface

## Completeness
- [ ] CHK001 - Are visual layout specifications defined for the sidebar component
  including width, collapse behavior, and resize constraints?
  [Completeness, Spec §FR-001]
- [ ] CHK002 - Are loading state behaviors documented for message streaming,
  including skeleton UI and progress indicators?
  [Completeness, Spec §FR-004]
- [ ] CHK003 - Are empty state designs specified for new conversations
  with no message history?
  [Completeness, Gap]

## Clarity
- [ ] CHK004 - Is the term "real-time response" quantified with specific
  latency thresholds (e.g., first token within Xms)?
  [Clarity, Spec §SC-001]
- [ ] CHK005 - Are the differences between "active conversation" and
  "selected conversation" terminology defined?
  [Clarity, Ambiguity]

## Consistency
- [ ] CHK006 - Are scroll behavior requirements consistent between
  the chat area (§FR-003) and sidebar conversation list (§FR-001)?
  [Consistency, Spec §FR-001/§FR-003]
- [ ] CHK007 - Are dark/light theme color token names consistent
  between spec and plan documents?
  [Consistency, plan.md §Design System]

## Scenario Coverage
- [ ] CHK008 - Are interaction patterns specified for both mouse
  and keyboard users across all interactive elements?
  [Scenario Coverage, Spec §FR-007]
- [ ] CHK009 - Are responsive breakpoint behaviors defined for
  sidebar visibility on narrow viewports?
  [Scenario Coverage, Gap]

## Edge Cases
- [ ] CHK010 - Are maximum message length display behaviors defined,
  including truncation and "show more" patterns?
  [Edge Cases, Spec §EC-002]
- [ ] CHK011 - Are behaviors specified for when streaming is interrupted
  mid-response (network drop, API timeout)?
  [Edge Cases, Spec §EC-001]

## Non-Functional
- [ ] CHK012 - Are accessibility requirements (WCAG level, screen reader support)
  documented for the chat interface?
  [Non-Functional, Gap]
- [ ] CHK013 - Are animation/transition performance budgets defined
  to prevent jank on lower-end devices?
  [Non-Functional, Gap]
참조 마커 해설

[Spec §FR-001] — spec.md의 특정 요구사항 참조
[Gap] — 명세에 아예 빠져있는 영역 발견
[Ambiguity] — 모호한 표현 발견
[Conflict] — 문서 간 충돌 발견

◆ 13. 도메인별 분리 생성

→ "성능 중심으로"

# 기존 ux.md에 CHK001~CHK013이 있는 상태에서 다시 실행하면
# 새 항목은 CHK014부터 시작하여 기존 파일 끝에 추가됨

◆ 14. 핵심 규칙

규칙 설명
80% 이상 추적성 체크리스트 항목의 80% 이상이 Spec §X.Y, Gap, Ambiguity, Conflict 등의 참조를 포함해야 함
구현 언어 금지 Verify, Test, Confirm, Click, Navigate, Render, Load, Execute 등의 동사 사용 불가. 오직 "Are... defined?", "Is... specified?" 패턴만 허용
기존 삭제 금지 이미 있는 체크리스트 항목은 절대 삭제하지 않음. 새 항목만 추가 (append only)
명세 검증만 시스템 동작이나 UI 렌더링이 아니라, 명세 문서 자체의 품질(완전성, 명확성, 일관성)을 검증
도메인별 파일 분리 짧고 설명적인 파일명 사용 (ux.md, security.md, performance.md). 하나의 파일에 모든 도메인을 섞지 않음

Implement 전 게이트로 활용

/speckit.implement 실행 시, checklists/ 폴더가 있으면 자동으로 완료율을 검증합니다.

# /speckit.implement 실행 시 자동 체크
Checklist Status:
  ux.md:          10/13 (76.9%) ⚠️
  security.md:     6/6  (100%) ✅
  performance.md:  7/8  (87.5%) ⚠️

⚠️ 일부 체크리스트의 완료율이 낮습니다. 계속 진행하시겠습니까?
체크리스트 항목 완료 방법

체크리스트의 - [ ] 항목은 해당 명세를 보완한 후 수동으로 - [x]로 변경합니다. 예를 들어 CHK003이 "빈 상태 디자인이 정의되었는가?"라면, spec.md에 빈 상태 디자인을 추가한 후 해당 항목을 체크합니다.

◆ 15. Analyze vs Checklist 비교

/speckit.analyze
목적문서 간 일관성 검증
관점spec ↔ plan ↔ tasks 교차 비교
출력Findings Table (문제 목록)
파일 수정안 함 (읽기 전용)
실행 시점Tasks 생성 후
반복수정 후 재검증
핵심 질문"문서들이 서로 맞는가?"
/speckit.checklist
목적단일 명세의 완성도 검증
관점명세 자체의 품질 평가
출력체크리스트 (검증 항목)
파일 수정checklists/ 파일 생성/추가
실행 시점Specify 완료 후 ~ Implement 전
반복도메인별 반복 생성
핵심 질문"명세가 충분히 좋은가?"
함께 사용하기

두 명령어는 상호 보완적입니다. Checklist로 명세 자체를 다듬고, Analyze로 문서 간 정합성을 검증하면 최고의 품질을 확보할 수 있습니다.

Checklist는 명세 작성 직후에 실행하여 Gap을 조기에 발견합니다.
Analyze는 모든 문서가 갖춰진 후에 실행하여 교차 정합성을 검증합니다.
두 도구 모두 구현 전에 사용하는 것이 핵심입니다.
